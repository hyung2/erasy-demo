# 안전도 점수 산식 SSOT v2 — 다차원 스코어링 엔진 (score-spec-v2-multidim)

> 엔진 v1(`score-spec-T3.0.md`, `lib/score.ts`)을 **폐기하고 처음부터 재설계**한 정본.
> v1 문서·코드는 보존한다(삭제 금지). 구현은 신규 `lib/score-v2.ts`로 하며, v2 구현·검증 완료 시점부터 이 문서가 산식 SSOT다.
> 상태: **설계 spec 완료 — 구현 미착수** (코드·시드·API 무변경). 구현은 11장 로드맵의 태스크로 분해되어 별도 착수한다.

---

## 1. 철학·목표 — 왜 다차원인가

### 1.1 "안전도"의 재정의

v1은 안전도를 "100에서 위험 신호만큼 깎은 잔여값" 하나로 정의했다. v2는 안전도를 **서로 다른 위협 질문 4개에 대한 서브 진단의 종합**으로 재정의한다:

> **안전도 = "지금 뚫려 있나(노출) × 관리 사각이 얼마나 넓나(표면) × 습관이 연쇄를 막나(위생) × 지금 공격받고 있나(위협)"에 대한 각각의 답 + 가장 약한 고리가 지배하는 종합.**

### 1.2 단일 감점 점수의 구조적 한계 (v1 폐기 사유)

1. **행동 지시 불가** — "52점"은 무엇부터 고칠지 말해주지 않는다. 북극성 지표(관심·재방문)가 되려면 점수가 다음 행동을 가리켜야 한다.
2. **상쇄(마스킹)** — 단일 합산에서는 위생이 처참해도 계정 수가 적으면 점수가 높게 나온다. 반대 방향도 성립한다. "유출된 비밀번호를 전 계정에서 재사용 중"인 사용자가 60점대를 받을 수 있는 구조는 위협 모델을 배반한다.
3. **worst-wins의 정보 소실** — v1은 이중계상을 피하려고 계정당 1티어(worst-wins)를 강제했다. 그 결과 "유출됐고 + 20개월 방치됐고 + 이상 접속까지 있는" 뽐뿌 계정의 신호 3개 중 2개가 점수에서 사라졌다. 다차원에서는 각 신호가 **자기 축**에서 온전히 계상되므로 worst-wins라는 억제 핵 자체가 불필요해진다.
4. **이질적 시간축 혼합** — 유출(사건, 해결 전까지 지속)·방치(누적, 느린 변화)·위생(습관, 행동 즉시 반영)·이상 접속(이벤트, 90일 감쇠)은 동역학이 다르다. 단일 스칼라는 이들의 변화 속도를 뭉개서 추이 그래프의 해석력을 죽인다.
5. **업계 선례 정합** — Microsoft Secure Score(카테고리별 세부 점수 + 종합), 보안 등급 서비스(BitSight·SecurityScorecard: 팩터별 등급 + 종합), 신용점수(요인별 기여 공개)가 모두 "서브 진단 + 종합" 구조다. Personal ASM의 소비자 버전으로서 같은 구조가 설득력·설명가능성 면에서 표준이다.

### 1.3 승계하는 v1 원칙 (재설계에서도 불변)

- **자격증명 무저장**: 입력 신호는 전부 메타(불리언·타임스탬프). 비밀번호 값·해시 미저장.
- **정직 표기**: coverage는 점수 보정이 아닌 병기. 모르는 것(미확인 신호)을 위험으로 단정하지 않는다.
- **규모 감점 배제**: 계정 수 자체는 감점하지 않는다. 위험 신호가 있는 계정만 계상.
- **단조성**: 위험 신호 추가는 점수를 올리지 않고, 회복 조치는 점수를 내리지 않는다.
- **데모 앵커 비의존**: 기존 62/52, Before/After 62→88에 맞춘 역산 캘리브레이션 금지. 모델이 산출하는 값을 새 앵커로 삼는다(사용자 확정 결정).

## 2. 축(dimension) 정의 — 위협모델 기반 4축

각 축은 0~100 서브점수(100 = 안전)이며, 사용자 언어로 된 질문 하나에 답한다.

| 축 | key | 축이 답하는 질문 | 위협 모델 | 시간축 |
|---|---|---|---|---|
| **유출 노출** (Exposure) | `exposure` | "내 정보가 이미 남의 손에 있나?" | 유출된 자격증명·PII → 계정 탈취·피싱 표적화. DBIR 2025: 탈취 자격증명 = 유출의 31%(직전 38%), 타 벡터의 2배+. 암시장 비밀번호 28억건 | 사건형 — 해결 전까지 지속 |
| **방치 표면** (Surface) | `surface` | "무감시 공격 표면이 얼마나 넓나?" | 휴면·미인지 계정은 유출 통지·이상 징후를 놓치는 사각. IBM 2024: 자격증명 벡터 탐지·차단 약 10개월 — 무감시가 이 지연의 본질. PIPC 크리덴셜 스터핑 6차례 적발 사례의 표적이 방치 계정 | 누적형 — 느린 변화 |
| **계정 위생** (Hygiene) | `hygiene` | "내 습관이 연쇄 피해를 막나?" | 재사용 비밀번호 1개 유출 → N계정 연쇄 탈취(크리덴셜 스터핑). DBIR 2025: 사용자별 고유 비밀번호 중앙값 49%뿐, SSO 로그인 시도의 일 중앙값 19%가 스터핑 | 습관형 — 행동 즉시 반영 |
| **진행형 위협** (Active Threat) | `threat` | "지금 공격받는 정황이 있나?" | 이상 접속 = 탈취 시도 진행 신호. 저빈도·고신호 — 발생 시 즉시성 최고 | 이벤트형 — 90일 창 |

**설계 노트 — 축 간 독립성과 "이중계상" 재해석**: 뽐뿌 계정(유출 + 20개월 방치 + 이상 접속)은 E·S·T 3개 축에 모두 계상된다. 이것은 이중계상이 아니라 **같은 계정을 3개의 다른 질문으로 본 것**이다. v1이 worst-wins로 억제했던 상관 신호 문제는, v2에서는 "종합 집계의 가중치가 축별 영향 총량을 구조적으로 제한"(5장)하는 방식으로 해소된다.

**대안 검토(채택하지 않은 축 구성)**:
- *3축안(E·S·H, T를 E에 흡수)*: 이상 접속은 "이미 노출됨"과 "지금 공격 중"이 사용자 행동(비번 교체 vs 세션 정리)이 달라 분리 유지가 낫다. 기각.
- *5축안(+프라이버시 발자국: 구독·PII 보유량)*: unsubscribe를 점수 신호로 승격하는 안. 현 데이터 모델에 PII 보유량 신호가 없어 MVP 범위 밖. 로드맵 후보로만 기록.

## 3. 산식 공통 원리 — 생존확률 곱셈 모델

### 3.1 축 내부 집계: 100 × ∏(1 − p_i)

각 축은 위험 신호 i마다 "그 신호 하나가 사고로 이어질 기여 확률" p_i를 부여하고, **아무 경로로도 뚫리지 않을 확률**을 점수로 삼는다:

```
axis = 100 × ∏ (1 − p_i)   // i = 해당 축의 활성 위험 신호(회복 반영 후)
```

**v1의 "가산 + 임의 캡" 대비 채택 근거**:
1. **자연 한계효용 체감** — n번째 동일 신호의 기여가 자동으로 줄어든다(0.95^6 = 0.735). v1의 캡(재사용 8, 2FA 4…)은 임의 상수였고 캡 도달 후 신호가 무시됐다. 곱셈 모델은 캡 없이 모든 신호가 항상 반영되면서 [0,100]에 유계.
2. **단조성 내장** — 신호 추가는 인자(1−p)<1 곱셈이므로 반드시 하락, 회복은 인자 제거이므로 반드시 상승.
3. **해석 가능** — "이 축이 버틸 확률"이라는 리스크 모델 표준 서사. p_i는 확률적 *스케일*로 캘리브레이션한 설계 파라미터이며 실측 확률이 아님을 문서·UI에서 과장하지 않는다(정직 표기).

### 3.2 계정 집계 방식 선택 근거 (worst / 평균 / 가중합 대비)

- *worst(v1 1층)*: 정보 소실 — 기각(2장).
- *단순 평균(비율)*: "위험 계정 3개/전체 30개 = 10%"는 계정을 늘리면 점수가 오르는 역인센티브. 표면·노출은 **절대량**이 위험이다 — 기각(단, 위생 축은 습관 지표라 비율이 옳다. 4.3 참조).
- *가중합+캡(v1 2층)*: 캡의 임의성 — 기각.
- **곱셈(채택)**: 위 근거. 축 성격에 따라 신호 단위는 계정(노출·표면·위협) 또는 비율(위생)로 다르게 두되, 유계·단조 프레임은 공통.

### 3.3 공통 규칙

- `removed`(delete/revoke done) 계정: 곱셈 축(E·S·T)에서 신호 제외(공격 표면 소멸). 총계·coverage에는 유지 — v1 승계. **비율 축(H)은 "중립 처리" 예외 적용 — 4.3 C2 절 참조** (순수 제외 시 안전 계정 삭제가 비율을 악화시키는 단조성 위반이 생기므로).
  - *알려진 단순화(정직 기록)*: 이미 유출된 PII는 계정을 삭제해도 회수되지 않는다. MVP는 v1 규칙(삭제 = 감점 전체 해제)을 승계하되, 실서비스 전 "PII 유출 인자는 삭제 후에도 50% 잔존" 같은 보수화가 재검토 대상. UI 카피는 "표면 제거"로 표현하고 "유출 무효화"로 과장하지 않는다.
- `lastUsedAt = null`(미확인): 해당 신호 미부여(무감점) + 축 coverage 하락 — v1 승계.
- 내부 계산은 raw float, 표시 시에만 round. 종합은 raw 축 점수로 계산.

## 4. 축별 산식

### 4.1 E — 유출 노출 (exposure)

**신호 매핑** (스키마: `Breach.resolved=false` ∧ 계정 미제거):

| 신호 | 판정 | p | 근거 |
|---|---|---|---|
| 유출 미해결 + 비밀번호 노출 | `exposedFields`에 "비밀번호" 포함 | **0.35** | 계정 탈취 직접 경로. DBIR 2025 탈취 자격증명 31% = 최대 단일 벡터, IBM 2024 초기 벡터 1위 16% |
| 유출 미해결 (PII만) | 비밀번호 미포함 | **0.20** | 피싱·스미싱 표적화(간접 경로). 인터파크 2016형(이메일·이름·주소) |

```
E = 100 × ∏(1 − p_breach)    // 미해결 유출 건별
```

**p 캘리브레이션 근거** (실측 확률이 아닌 심각도 사다리 설계): 비밀번호 유출 1건 단독 → E=65(주의권 진입), 2건 → 42, 3건 → 27(위험 확정). "미해결 비밀번호 유출은 1건부터 주의, 3건이면 위험"이라는 도메인 판단을 역산한 스케일. 직접 경로(0.35) : 간접 경로(0.20) 비율은 DBIR의 "탈취 자격증명 = 타 벡터 2배+" 서열을 반영.

**회복**: `Breach.resolved=true` → 해당 인자 제거. `severity`는 산식 미사용(exposedFields가 핵심 구분을 이미 담당 — YAGNI), 표시·정렬용으로만.

**대안(기각)**: 유출 경과연수 감쇠 — 자격증명은 수년간 암시장에서 유통(2015 뽐뿌 비밀번호가 2026년에도 유효한 것이 본 데모의 실화 기반 서사)되므로 시간 감쇠 없음이 옳다.

### 4.2 S — 방치 표면 (surface)

**신호 매핑** (계정별, 미제거 계정만):

| 신호 | 판정 | p | 근거 |
|---|---|---|---|
| 휴면 24개월+ | `lastUsedDays >= 730` | **0.10** | 완전 무감시 표면 — 유출 통지·이상 징후를 놓칠 확률 최대. PIPC 스터핑 적발 사례(섹타나인·인터파크·워크넷)의 전형 표적 |
| 휴면 12~24개월 | `lastUsedDays >= 365` | **0.05** | 방치 초기 |
| 해외 + 활성(<12개월) | `category='overseas'` ∧ 확인됨 | **0.02** | 관할권 마찰(삭제·구제 지연)만 — 활성 관리 중이므로 최소 |
| 미인지 계정 | `discovered=true` ∧ 미조치(제거·해결 이력 없음) | **0.03** | "몰랐던 계정" = 존재 자체가 관리 사각. 휴면 인자와 독립 곱(몰랐고 + 방치면 둘 다) |

```
S = 100 × ∏(1 − p_account_signal)    // 한 계정이 휴면+미인지면 두 인자 모두 곱함
```

휴면 3단계는 상호배타(계정당 최심 1개), 미인지·해외는 독립 인자. 캘리브레이션: 24개월+ 휴면 5개 방치 → S=59(주의·위험 경계) — 방치 포트폴리오가 종합을 끌어내리도록(5장 블렌드 경유) 설계.

**회복**: delete/revoke done → 인자 제거. 재사용(재로그인)으로 `lastUsedAt` 갱신 → 자연 회복.

### 4.3 H — 계정 위생 (hygiene)

유일한 **비율 기반** 축 — 위생은 절대량이 아니라 습관의 질이므로, 해당되는 계정 대비 비율이 옳다(계정이 많다고 습관이 나빠지지 않는다).

**모수(분모) 정의**:
- `passwordHolders` = **위생 신호가 관측된** manual 계정 ∪ 비밀번호 신호가 관측된 계정(`passwordReused=true` 이력 또는 `password_change` done 이력 보유). 비밀번호 존재를 아는 계정만 분모에 — 모르는 계정을 분모에 넣어 비율을 희석하지 않는다(정직 표기). 분모는 회복 후에도 유지(교체해도 "비밀번호 보유 계정"임은 불변 — 비율 회복은 분자에서).
- `manualAccounts` = **위생 신호가 관측된** `provider='manual'` 계정. OAuth 계정의 2FA는 provider 위임이라 제외 — v1 결정 승계.

**[관측 게이트] `passwordSignalObserved` (2026-07-23 추가)**: 위 두 모수는 `passwordSignalObserved=true`인 계정만 편입한다.
- 시드·OAuth 연동 계정(`source ∈ {seed, oauth_linked}`) = true. 수집 경로가 존재한다.
- 사용자가 서비스명만 적어 직접 추가한 계정(`source=user_input`) = 재사용·2FA 중 하나라도 신고했을 때만 true.
- **근거**: 직접추가 API는 로그인 방식을 알 수 없어 `provider`를 `manual`로 기본 지정한다. 게이트가 없으면 이 계정이 "고유 비밀번호를 쓰고 2FA는 없는 계정"으로 두 모수에 동시 편입돼, 재사용률(가중 0.65)이 희석되는 폭이 2FA율(0.35) 하락보다 커서 **위생 점수가 오른다**. 실측(2026-07-23, 24계정 인벤토리 기준): 계정 1개 추가로 hygiene 8.8 → 16.3, 최약축이라 종합 24 → 29. "몰랐던 계정을 발견할수록 안전도가 좋아지는" 결과는 1.3 정직 표기 원칙(모르는 것을 단정하지 않는다)을 반대 방향으로 위반한다.
- **잔여 한계**: 사용자가 "재사용 안 함·2FA 없음"을 모두 '아니오'로 신고한 경우, 현 스키마에 신고 여부 자체를 남기는 필드가 없어 미관측과 구분되지 않는다. 이때 2FA 부재라는 위험 신호가 계상되지 않아 점수가 다소 높게 나온다. 신고 시각 필드 추가(마이그레이션 1건)로 해소 가능 — 데모 이후 과제.

```
reuseRate = 미해결 재사용 계정 수(passwordReused ∧ !passwordChanged) / |passwordHolders|
tfaRate   = 2FA 설정 manual 계정 수 / |manualAccounts|

H = 100 × ( 0.65 × (1 − reuseRate) + 0.35 × tfaRate )
```

**가중 0.65 / 0.35 근거**: 재사용은 유출 1건을 N계정 연쇄로 **증폭**시키는 인자(DBIR 2025: 고유 비밀번호 중앙값 49%뿐, 스터핑이 SSO 로그인 시도의 19%) — 킬체인의 본체. 2FA는 그 연쇄를 끊는 **완화 계층** — 중요하되 종속 변수. 약 2:1.

**미측정 처리**: `passwordHolders`=0 → reuse 항 미측정, `manualAccounts`=0 → 2FA 항 미측정. 한쪽만 미측정이면 측정된 항 100% 가중으로 재정규화, 양쪽 다면 **축 자체 미측정**(`measured:false`, 5.3 재정규화) — 모르는 위생을 0점도 100점도 아닌 "확인 불가"로.

**회복**: `password_change` done → 분자에서 제외(reuseRate 하락). 2FA 설정(`twoFactorEnabled` 갱신) → tfaRate 상승.

**삭제 계정의 H 처리 — "중립 처리" (C2 단조성 게이트, 2026-07-14 확정)**:

비율 축에서 removed 계정을 분모·분자에서 순수 제외하면 **안전한 계정(고유 비밀번호·2FA 설정) 삭제 시 비율이 하락**하는 단조성 위반이 생긴다(레드팀 C2: 2FA manual 계정 삭제 → tfaRate 하락 → H 하락 → expectedGain 음수 −10.25 재현). 이를 막기 위해 removed 계정은 H에서 다음과 같이 **비대칭 처리**한다:

- **위험 신호 보유 계정**(미해결 재사용 / 비2FA manual) 삭제 → 해당 신호와 함께 분모·분자에서 **완전 제외** — 위험 표면이 사라졌으므로 비율 개선 (예: 워크스루 계단 4 — 비2FA 뽐뿌 삭제 → tfaRate 0.25→0.333 상승)
- **안전 계정**(고유 비밀번호 / 2FA 설정) 삭제 → 분모·분자 **신용 유지(중립)** — 좋은 습관의 증거는 계정 삭제로 소멸하지 않으므로 H 불변 (예: 2FA 토스 삭제 → H 유지)

이중 방어(belt): ① expectedGain은 `max(0, simulated − current)` clamp — 어떤 회복 액션도 음수 이득 표시 불가 ② 추천기는 weakestAxis=hygiene일 때 delete가 아닌 `password_change`를 고정 추천 — 위생 개선 목적의 삭제 추천 자체를 차단.

*설계 근거*: "삭제 = 표면 제거"(E·S·T 개선)와 "위생 = 습관의 질"(H)은 다른 차원이다. 계정을 지웠다고 습관이 나빠지지 않고, 좋은 습관 계정을 지웠다고 습관이 좋아지지도 않는다 — 중립이 도메인 정합. 워크스루 전 숫자 재현 + raw H 단조성 보장을 fixture로 검증(scripts/score-v2-fixture.test.ts C2 3종).

### 4.4 T — 진행형 위협 (threat)

**신호 매핑**: 최근 90일 `AccessLog.suspicious=true` 보유 계정(세션 정리 미완, 미제거).

```
T = 100 × 0.60^k    // k = 이상 접속 계정 수 (p = 0.40)
```

**p=0.40(고단가) 근거**: 진행형 위협은 저빈도·고신호 — 이상 접속 1건은 "확률적 위험"이 아니라 "탈취 시도 관측"이다. 1건만으로 T=60까지 급락시켜 축이 비명을 지르게 한다. 정적 신호(E·S·H)와 달리 즉시 행동(세션 정리·비번 교체)을 트리거하는 축이므로 민감도가 곧 기능이다.

**회복**: `logout_sessions` done → 해당 계정 인자 제거. 90일 창 경과 → 자연 소멸(윈도우가 시간 감쇠를 겸함).

**coverage 특칙**: T는 접속기록이 관측 가능한 계정에만 성립. `AccessLog`가 1건이라도 있는 계정을 관측 모수로 삼아 "관측 가능 N개 계정 기준" 병기. 관측 0이면 축 미측정(무감점) — 미관측을 위험으로 단정하지 않는다.

## 5. 종합 집계 — 가중평균·최약축 블렌드

### 5.1 산식

```
WA        = Σ w_i × axis_i          // 가중평균
worst     = min(axis_i)             // 측정된 축 중 최저
composite = clamp( round( (1−λ) × WA + λ × worst ), 0, 100 )

w = { exposure: 0.35, hygiene: 0.30, surface: 0.20, threat: 0.15 },  λ = 0.45
```

### 5.2 집계 방식 선택 근거 (대안 비교)

| 방식 | 판정 | 사유 |
|---|---|---|
| 가중평균 단독 | 기각 | 마스킹 — 한 축 0점이어도 나머지가 가려줌(위생 0 + 나머지 100 → 70점 "주의" 오판) |
| 최악축 단독(min) | 기각 | 다른 축 개선이 전혀 반영 안 됨 — 회복 아크·재방문 유인 사망 |
| 조화평균 | 기각 | 저축 지배 성질은 좋으나 축=0에서 발산(epsilon 땜질 필요), 심사·사용자에게 설명 불가 |
| 최악축 캡(min(WA, worst+M)) | 기각 | 최약축이 안 움직이는 동안 헤드라인이 동결 — 본 데모 시드로 실검산 결과 스냅샷 3개가 전부 동일값이 되어 추이 서사 소멸 |
| **블렌드(채택)** | **채택** | 한 문장 설명 가능("종합 = 평균 55% + 가장 약한 고리 45%"), 모든 축 변화가 헤드라인에 반영(추이 생존), 최약축이 지배적으로 끌어내림(마스킹 차단) |

### 5.3 성질 (fixture 검증 대상)

1. **마스킹 차단**: 임의 한 축=0, 나머지 전부 100일 때 최악 케이스 WA=85(최저 가중 0.15 축이 0일 때) → composite = 0.55×85 = 46.75 < 50 → 항상 "위험". **λ=0.45는 이 성질을 보장하는 최소값(λ > 0.412)에서 여유를 둔 채택값**.
2. **합의 보존**: 전 축 동일 점수 x → composite = x (블렌드 항등).
3. **단조성**: 축 점수는 신호에 단조(3.1), WA·min 모두 축에 단조 → composite 단조.
4. **미측정 재정규화**: 미측정 축 제외 후 남은 축 가중치를 합 1로 재정규화, worst는 측정 축에서만. 측정 축 1개뿐이면 composite = 그 축. 측정 축 0개(계정 0) → 점수 없음("측정 불가" 상태 반환, 0점 아님).

### 5.4 축 가중치 w 근거

- **E 0.35 + H 0.30 = 0.65**: 자격증명 킬체인(유출된 비밀번호 × 재사용)이 유출 경로의 압도적 1위 — DBIR 31%(타 벡터 2배+), IBM 초기 벡터 1위 16%. 노출(이미 발생)이 위생(증폭 계수)보다 반 발 앞.
- **S 0.20**: 방치는 직접 경로가 아닌 **enabler**(탐지 지연·통지 상실) — IBM "약 10개월 탐지·차단"의 개인 버전. PIPC 적발 사례들의 공통 표적이나 그 자체로 탈취는 아님.
- **T 0.15**: 저빈도라 상시 가중은 낮게, 대신 발생 시 축 내 단가(p=0.40)가 커서 실질 영향은 급등 — "평시 조용, 유사시 비명" 설계.

가중치·p·λ는 근거 데이터로 *서열과 스케일*을 정당화한 설계 캘리브레이션이며 실측 적합값이 아니다 → **교차검증 플래그**(13장).

## 6. 등급·해석·행동 유도

### 6.1 등급 밴드 (v1 유지 — FE `deriveGrade` 호환)

| 등급 | 범위 | 의미 |
|---|---|---|
| 양호 | 80~100 | 전 축 관리권(블렌드 특성상 최약축 ≥ 약 55 필요) |
| 주의 | 50~79 | 최소 1개 축에 실질 결함 |
| 위험 | 0~49 | 최약축 심각 또는 다축 결함 — 단일 축 전멸 시 무조건 이 밴드(5.3-1) |

4등급 분할(양호/보통/주의/위험)은 FE·카피 전면 수정 대비 이득이 불명확해 보류 — 열린 결정(13장).

### 6.2 "가장 먼저 고칠 축" 도출

```
weakestAxis = argmin(측정된 axis_i)     // 동률 시 가중치 w 큰 축 우선
```

축 → 우선 액션 → SEE/CUT/GUARD 매핑:

| 최약축 | 우선 액션(actionType) | 원칙 | 추천 카피 골격 |
|---|---|---|---|
| hygiene | `password_change`(재사용 계정) + 2FA 안내 | CUT→GUARD | "유출된 비밀번호를 N개 계정에서 재사용 중 — 교체가 가장 급해요" |
| exposure | 유출 해결 플로우(비번 교체 후 resolved 마킹) | CUT | "미해결 유출 N건 — 여기가 뚫린 문이에요" |
| surface | `delete`/`revoke`(휴면·미인지 계정) | CUT | "N개 계정이 무감시 상태 — 안 쓰면 지워요" |
| threat | `logout_sessions` + 비번 교체 병행 안내 | GUARD | "이상 접속 정황 — 지금 세션부터 끊어요" |

SEE는 특정 축이 아니라 **축별 진단 화면 자체**(4축 카드/레이더)가 담당한다.

### 6.3 expectedGain — 행동의 가치를 숫자로

엔진은 순수 함수이므로 "이 액션을 완료했다고 가정한 가상 상태"로 재계산한 종합 delta를 **액션 버튼 옆에 병기**할 수 있다:

```
expectedGain(action) = composite(rows ⊕ action적용) − composite(rows)
```

김민준 기준 "비밀번호 교체 완료 시 **+22점**"(9.3) — 재방문·실행 전환의 핵심 훅. 계산 비용 무시 가능(순수 함수 재호출). 이것이 다차원 전환의 **비대칭 upside**: 단일 점수에서는 불가능했던 "액션별 가격표"가 구조적으로 공짜로 나온다.

|---|---|---|---|
| exposure | Quora(비번 0.35)·뽐뿌(비번 0.35)·인터파크(PII 0.20): 100×0.65×0.65×0.80 | 33.8 | **34** |
| surface | 휴면12~24개월 6개(카카오스토리·Spotify·Amazon·Medium·Quora·뽐뿌)×0.95 · 해외활성 2개(Netflix·Notion)×0.98 · 미인지 미조치 2개(Quora·뽐뿌)×0.97 · 싸이월드는 removed 제외: 100×0.95⁶×0.98²×0.97² | 66.43 | **66** |
| hygiene | passwordHolders 6(manual 4 + Gmail·네이버), 재사용 6/6=100% · 2FA manual 1/4=25%: 100×(0.65×0 + 0.35×0.25) | 8.75 | **9** |
| threat | 뽐뿌 이상접속 1건(미정리): 100×0.60 | 60.0 | **60** |

### 9.2 종합 = **24 (위험)**

```
WA    = 0.35×33.8 + 0.20×66.43 + 0.30×8.75 + 0.15×60.0 = 36.74
worst = hygiene 8.75
종합  = 0.55×36.74 + 0.45×8.75 = 24.15 → 24
```

- **weakestAxis = hygiene(9)** → 첫 추천 액션 = 비밀번호 교체. 서사 정합: 김민준의 실제 킬체인은 "2015 뽐뿌 유출 비밀번호를 6계정에서 재사용" — DBIR 1위 벡터 그대로.
- **기존 앵커 대체: 62(v0)/52(v1) → 24(v2, 위험)**. 하락 이유는 엔진이 더 정직해졌기 때문: 미해결 유출 3건 + 재사용률 100% + 이상 접속 진행이 "주의"일 수 없다는 위협 모델 판단. 데모 심사 관점에서도 "위험 → 양호" 아크가 "주의 → 양호"보다 강하다.

### 9.3 스냅샷 이력·회복 계단 (전부 엔진 실계산, 임의값 없음)

**이력** (시드 스냅샷 2건 대체값):

| 시점 | 상태 | 축(E/S/H/T) | 종합 |
|---|---|---|---|
| 8일 전 | 싸이월드 회복 전(휴면24m+·미인지 3개), 이상접속 없음 | 33.8 / 57.99 / 8.75 / 100 | **27** |
| 3일 전 | + 뽐뿌 이상접속 | 33.8 / 57.99 / 8.75 / 60 | **23** |
| 현재 | + 싸이월드 revoke done | 33.8 / 66.43 / 8.75 / 60 | **24** |

trend [27, 23, 24] · delta +1 — "이상 접속으로 하락 → 첫 정리로 소폭 회복, 그러나 진짜 레버(비밀번호)는 아직"이라는 v1보다 정확한 서사. 헤드라인이 +1에 그치는 것 자체가 제품 메시지다: *"하나 지웠다고 안전해지지 않아요."*

**회복 계단** (데모 Before/After 연출의 실계산 근거):

| 단계 | 액션 | 원칙 | 축(E/S/H/T) | 종합 | delta |
|---|---|---|---|---|---|
| 0 현재 | — | SEE(진단) | 33.8/66.4/8.8/60 | **24** | — |
| 1 | 재사용 6계정 비밀번호 교체 | CUT | 33.8/66.4/73.8/60 | **46** | +22 |
| 2 | 유출 3건 해결(resolved) | CUT | 100/66.4/73.8/60 | **71** | +25 |
| 3 | 뽐뿌 세션 일괄 정리 | GUARD | 100/66.4/73.8/100 | **77** | +6 |
| 4 | Quora·뽐뿌·Medium delete + 카카오스토리 revoke | CUT | 100/86.7/76.7/100 | **84** | +7 |
| 5 | Amazon·인터파크 2FA 설정 | GUARD | 100/86.7/100/100 | **93** | +9 |

- **Before/After 대체값: 62→88 → 24→93** (또는 데모 중간 컷이면 24→71). `targetScore` 85 → **90** 교체 제안.
- 계단의 delta 크기 순서(+25 유출해결, +22 비번교체 ≫ +6~9)가 근거 데이터의 위협 서열과 일치 — "점수가 위협 모델을 그대로 말한다"가 심사 소구점.
- 페르소나 3인 배선: 지민 SEE(단계 0 진단·4축 카드), 수현 CUT(단계 1·2·4), 재현 GUARD(단계 3·5 + 알림). 페르소나 문서·데모 대본 갱신은 후속(사용자 확정 사항).

## 10. 구현 영향 분석 (v1 대비)

### 10.1 API DTO — additive 확장 (breaking change 없음)

기존 `ScoreDTO` 6필드는 전부 유지(값 의미: score=종합, coverage=surface축 대표값). 아래를 추가:

```ts
export type AxisKey = 'exposure' | 'surface' | 'hygiene' | 'threat';

export type AxisDTO = {
  key: AxisKey;
  label: string;                 // "유출 노출" 등
  score: number | null;          // null = 미측정
  measured: boolean;
  coverage: number;              // 0~1
  coveredCount: number;
  totalCount: number;
  topFinding: string | null;     // 대표 근거 1줄 ("유출된 비밀번호를 6개 계정에서 재사용 중")
};

export type RecommendedActionDTO = {
  axis: AxisKey;
  actionType: 'password_change' | 'delete' | 'revoke' | 'logout_sessions' | 'resolve_breach' | 'enable_2fa';
  accountIds: string[];
  expectedGain: number;          // 완료 가정 시 종합 delta (6.3)
};

// ScoreDTO에 추가되는 필드
axes: AxisDTO[];
weakestAxis: AxisKey;
recommendedAction: RecommendedActionDTO | null;
```

주: `resolve_breach`·`enable_2fa`는 CleanupRequest actionType이 아닌 추천 시맨틱(기존 플로우·신호 갱신으로 처리) — enum 확장 없이 DTO 레벨 표현.

### 10.2 스키마·시드·마이그레이션

| 항목 | 판정 |
|---|---|
| Prisma 스키마 | **변경 불필요** — 4축 입력 신호 전부 기존 컬럼으로 충족(lastUsedAt·twoFactorEnabled·passwordReused·discovered·Breach.*·AccessLog.suspicious·CleanupRequest.*) |
| 마이그레이션 | **0건** (축별 추이 저장을 채택할 경우에만 `ScoreSnapshot.axes Json?` 1건 — 열린 결정) |
| 시드 데이터 | **신호 무변경**. 스냅샷 2건은 이미 `historicScore()`로 엔진 재계산하므로 v2 엔진 연결만으로 47/50 → 23/27 자동 갱신 |
| dummy-data | `derivePrivacyScore`(62 산식)는 FE 잔존 참조용 — FE 배선 교체 시 참조 제거 대상. `targetScore` 85→90 갱신 |

### 10.3 코드 구조

- `lib/score-v2.ts` **신규**(v1 `score.ts` 보존·미수정 — 파일 보존 원칙). 순수 함수: `AccountSignalRow[]`(v1 타입 재사용 가능) → `{ composite, axes[], weakestAxis, recommendedAction }`.
- `lib/score-service.ts` 엔진 호출부를 v2로 전환 + DTO 조립 확장. 폴백 서열(실계정→시드→메모리)·스냅샷 정책 유지.
- `computePrivacyScore` v1 시그니처는 v1 파일에 그대로 잔존(참조 이력·fixture 보존).

### 10.4 FE 영향 + 데모 단순성 완충 UX

현 dashboard는 dummy-data 직배선 상태(API 배선 T3.4 미착수)라 **재설계 최적 타이밍** — 낡은 계약에 배선했다 다시 뜯는 비용이 없다.

**완충 원칙: 헤드라인은 v1과 동일하게 단순, 깊이는 opt-in.**
1. **레이어 1(기본 노출)**: 종합 큰 숫자 + 등급 + delta + coverage 병기 — 기존 화면과 시각적으로 동일. 심사·시연의 3초 이해를 보존.
2. **레이어 2(펼침/스크롤)**: 4축 카드(점수 게이지 + topFinding 1줄 + 추천 액션 버튼 + expectedGain "+22"). 대안 시각화: 4축 레이더(면적=안전) — 카드가 액션 연결에 유리해 기본은 카드, 레이더는 결과 화면 Before/After 오버레이 연출용.
3. **결과 화면(/cleanup/result)**: 종합 상승 애니메이션(기존) + 축별 delta 바("위생 9→74") — 계단 서사 시각화.
4. **한 줄 요약 카피**: "가장 약한 고리: 계정 위생 9점" — 다차원의 복잡도를 문장 하나로 접는 장치.

영향 화면: dashboard(점수 홈)·cleanup/result·guard(알림 카피에 축 언급). accounts·access는 무영향.

## 12. 검증 (DoD 총괄)

1. **앵커 재현**: 현 시드 → 종합 24, 축 [34, 66, 9, 60], trend [27, 23, 24].

## 13. 열린 결정 — 사용자 컨펌 필요 지점

1. **앵커 하락 수용**: 김민준 현재 52(주의) → **24(위험)**. 데모 서사는 강해지나 "시작이 위험"이 심사 톤에 맞는지 최종 판단 필요. (완화 옵션: exposure p를 0.30/0.15로 낮추면 현재 약 27~29 — 여전히 위험 밴드.)
2. **파라미터 교차검증**: 축 가중치 w·신호 p·λ=0.45는 근거 데이터 기반 설계 캘리브레이션이며 실측 적합이 아님 — 구현 전 다른 LLM 교차검증 + 업계 스코어링(Microsoft Secure Score 등) 대조 1패스 권장.
3. **축별 추이 저장**: MVP는 종합만 스냅샷(마이그레이션 0건) vs `axes Json?` 추가(1건) — 데모에 축별 추이 그래프가 필요한지로 결정.
4. **등급 3밴드 유지 vs 4밴드** — 기본 제안은 유지.
5. **discovered 신호의 surface 축 포함(p=0.03)** — 제외 시 현재 surface 66→71, 종합 미세 상승.
6. **targetScore 85 → 90** 및 데모 중간 컷 지점(계단 어디까지 시연할지).
