# 프라이버시 점수 산식 SSOT — score-spec-T3.0 (엔진 v1)

> W3 T3.0 산출물. `lib/score.ts`(순수 엔진) · `lib/score-service.ts`(DB 집계) · `prisma/seed.ts`(신호 주입) · `GET /api/score`의 정본.
> 코드와 이 문서가 어긋나면 이 문서가 우선하며, 코드 수정으로 정합시킨다.
> 형제 문서: `../03-step02-mvp/`(기획 루트)의 T1.2 아키텍처 락, T1.4 스키마 초안과 신호 정의를 공유한다.

## 1. 설계 목표

- **앵커 맞춤 캘리브레이션이 아니라 도메인 논리 우선**: 신호별 가중치는 "62점 재현"이 아니라 위협 모델(계정 탈취·크리덴셜 스터핑·방치 표면)에서 도출한다. 재계산 결과 앵커가 이동하는 것을 허용한다(사용자 확정 결정).
- **자격증명 무저장 원칙**: 모든 입력 신호는 비밀번호 값과 무관한 메타 신호(불리언·타임스탬프)만 사용한다(T1.4).
- **정직 표기(O5)**: coverage는 점수를 보정하지 않고 "확인된 N개 계정 기준"으로 병기만 한다. 미확인 신호에는 감점하지 않는다(모르는 것을 위험으로 단정하지 않음).
- **단조성**: 위험 신호 추가는 점수를 올리지 않고, 회복 조치는 점수를 내리지 않는다(fixture로 검증).

## 2. 산식

```
score = clamp( round( 100 − Σ(1층 주신호 감점) − Σ(2층 보조신호 가산, 캡 적용) ), 0, 100 )
```

### 2.1 1층 주신호 — 계정당 1개 티어 (상호배타, worst-wins)

한 계정이 여러 신호를 가져도 가장 심한 티어 1개만 감점한다. 유출·휴면·해외는 강하게 상관된 신호(방치된 해외 계정이 유출되는 패턴)라 중복 감점하면 같은 위험을 이중 계상하기 때문.

| 티어 | 조건 | 감점 | 근거 |
|---|---|---|---|
| P1a | 유출 미해결 + **비밀번호 노출**(`exposedFields`에 "비밀번호") | **−9** | 계정 탈취 직접 경로. 즉시 조치 대상 최상위 |
| P1b | 유출 미해결(PII만, 비밀번호 미노출) | **−6** | 피싱·스미싱 표적화 위험. 탈취 직접 경로보다는 낮음 |
| P2 | 휴면 24개월+ (비유출) | **−5** | 무감시 공격 표면. 유출 통지·이상 징후를 놓칠 확률 최대 |
| P3 | 휴면 12~24개월 (비유출) | **−2** | 방치 초기 단계 |
| P4 | 해외 서비스 + 활성(휴면 12개월 미만, 비유출) | **−1** | 관할권 이슈(삭제·구제 마찰)만. 활성 관리 중이므로 최소 |
| P0 | 그 외 (활성·국내·소셜) | 0 | 위험 신호 없음 — 계정 수 자체는 감점하지 않음(규모 감점 배제, 사용자 결정) |

초안 대비 정정: 초안은 "비유출·방치 −8 > 유출 미해결 −6"으로 휴면이 유출보다 무거웠다. 위협 모델상 미해결 유출(특히 비밀번호 노출)이 방치보다 명백히 위험하므로 서열을 P1 > P2로 교정했다. 초안의 서열은 dummy 시연값(62) 역산의 흔적으로 판단.

- 휴면 경계: `lastUsedAt` 기준 730일(24개월) / 365일(12개월).
- `lastUsedAt = null`(미확인): 티어 미부여(감점 0) + coverage 분모에서 "미확인"으로 집계. ※ `/api/accounts`의 표시용 폴백(null→3650일 취급)과 다름 — 엔진은 미확인 무감점이 정본. T3.4 배선 시 정합 검토 포인트.

### 2.2 2층 보조신호 — 계정 횡단 가산 (합산 후 캡)

1층과 독립인 위생(hygiene) 신호. 유출된 계정의 비밀번호 재사용은 **다른** 계정까지 위험하게 하므로(크리덴셜 스터핑) 1층과 별도로 가산하되, 한계효용 체감(같은 크리덴셜 세트의 n번째 계정)을 캡으로 반영한다.

| 신호 | 가산 | 캡 | 근거 |
|---|---|---|---|
| 비밀번호 재사용 (`passwordReused`) | +2/계정 | 10 → **8** | 스터핑 증폭 계수. 캡 8 = 재사용 4계정 이후 한계효용 0 |
| 2FA 미설정 — **manual 계정 한정** (`provider='manual' && !twoFactorEnabled`) | +1/계정 | **4** | OAuth 계정 2FA는 provider 위임이라 전수 감점 왜곡 → manual만 |
| 이상 접속 — 최근 90일 `AccessLog.suspicious` 보유 계정 | +3/계정 | **6** | 진행형 위협(탈취 시도 정황). 정적 신호보다 단가 높음 |

### 2.3 회복 규칙 (감점 해제)

| 조치 | 효과 |
|---|---|
| `Breach.resolved = true` | 해당 유출 티어 제외 → 계정은 하위 티어(P2~P0)로 재평가 |
| `CleanupRequest done` — `delete`/`revoke` | 해당 계정 전체 감점 제외(공격 표면 제거). 계정은 총계·coverage에는 유지 |
| `CleanupRequest done` — `password_change` | 해당 계정의 재사용 가산 해제 |
| `CleanupRequest done` — `logout_sessions` | 해당 계정의 이상 접속 가산 해제 |
| `CleanupRequest done` — `unsubscribe` | 점수 무영향(구독 정리는 비용 신호, 프라이버시 신호 아님) |

- 재계산마다 `ScoreSnapshot` 조건부 append: 최신 스냅샷과 점수가 다르거나, 최신이 24시간 이전이면 기록(동일 점수 반복 조회 시 행 남발 방지). 하드코딩 `scoreTrend` 대체.
- `delta` = 현재 점수 − 직전 스냅샷 점수. `trend` = 최근 스냅샷 최대 6개(시간 오름차순).

### 2.4 coverage (O5 정직 표기)

```
covered = lastUsedAt이 확인된 계정 수
coverage = covered / totalAccounts
```

점수 보정 계수가 아니다. UI는 "확인된 {covered}개 계정 기준" 병기. 시드 24계정은 전부 확인 → 1.0. W4 직접입력(F2) 계정이 `lastUsedAt` 없이 들어오면 coverage가 자연 하락한다.

## 3. 시드 신호 정의 (T3.2 — 수현/김민준 페르소나 서사)

> 표시 이름은 기존 데모 정본(김민준) 유지. 서사 골격: "2015년 뽐뿌에서 유출된 비밀번호를 아직 6개 계정에서 재사용 중 + 방치 계정에서 이상 접속 발생 + 어제 첫 정리 완료".

- **재사용 6계정** (비밀번호 보유 계정만 — manual 4 + 루트 provider 계정 2): Gmail(a01)·Amazon(a12)·인터파크(a16)·뽐뿌(a17)·네이버(a19)·토스(a22). `reuseGroupId`는 T1.4 권고대로 MVP 미사용(불리언만).
- **2FA 설정**: Gmail·카카오톡·iCloud·토스 true. → manual 중 2FA 미설정 = Amazon·인터파크·뽐뿌 3계정.
- **이상 접속**: 뽐뿌(a17)에 3일 전 위치 미상 suspicious 로그 1건(20개월 미사용 계정에 제3자 접속 시도 정황). 그 외 정상 로그 4건.
- **discovered**(몰랐던 계정): Quora(a15)·뽐뿌(a17)·싸이월드(a18) — "이 중 3개는 모르셨죠".
- **riskTags**: 신호에서 파생(breach/reuse/dormant/no2fa) + 구독 서비스(Netflix·Spotify·Apple Music·멜론)에 subscription.
- **CleanupRequest**: dummy `deleteRequests` 정합 — 싸이월드 revoke **done**(어제), Quora delete in_progress, Medium delete queued, 카카오스토리 revoke queued.
- **ScoreSnapshot 이력 2건**: 8일 전 50(이상접속 발생 전), 3일 전 47(이상접속 반영). 각 시점의 엔진 계산값과 수학적으로 일치(임의값 아님 — 4.2 참조).

## 4. 앵커 정합 계산표 (현 시드 기준)

### 4.1 현재 점수 = **52 (주의)**

1층 (worst-wins):

| 계정 | 신호 | 티어 | 감점 |
|---|---|---|---|
| a15 Quora | 유출 미해결(이메일+비밀번호), 해외, 14개월 휴면 | P1a | −9 |
| a17 뽐뿌 | 유출 미해결(비밀번호), 20개월 휴면 | P1a | −9 |
| a16 인터파크 | 유출 미해결(이메일·이름·주소), 8개월 | P1b | −6 |
| a18 싸이월드 | 36개월 휴면 → **revoke done(회복)** | 제외 | 0 |
| a07 카카오스토리 18m / a11 Spotify 13m / a12 Amazon 12m / a14 Medium 15m | 휴면 12~24개월 | P3 ×4 | −8 |
| a10 Netflix / a13 Notion | 해외 + 활성 | P4 ×2 | −2 |
| 나머지 14계정 | 활성·국내·소셜 | P0 | 0 |
| **1층 소계** | | | **−34** |

2층:

| 신호 | 계정 | 계산 | 가산 |
|---|---|---|---|
| 재사용 | 6계정 | min(6×2, 8) | +8 |
| 2FA 미설정(manual) | Amazon·인터파크·뽐뿌 | min(3×1, 4) | +3 |
| 이상 접속(90일) | 뽐뿌 | min(1×3, 6) | +3 |
| **2층 소계** | | | **+14** |

**score = 100 − 34 − 14 = 52** · grade 주의(50~79) · coverage 24/24 = 1.0

### 4.2 스냅샷 이력 정합

| 시점 | 상태 차이 | 계산 | 점수 |
|---|---|---|---|
| 8일 전 | 싸이월드 회복 전(P2 −5 포함), 이상접속 없음 | 100 − 39 − 11 | **50** |
| 3일 전 | + 뽐뿌 이상접속(+3) | 100 − 39 − 14 | **47** |
| 현재 | + 싸이월드 revoke done(+5 회복) | 100 − 34 − 14 | **52** |

trend [50, 47, 52] · delta +5 — "이상 접속으로 하락 → 첫 정리로 회복"의 실계산 서사.

### 4.3 앵커 이동 보고 (62 → 52)

- 기존 62는 dummy 시연 가중치(유출 6·방치 8·중위험 2)의 산물. 엔진 v1은 유출>방치 서열 교정 + 재사용·2FA·이상접속 신호 추가 + 회복 규칙 반영으로 **52(주의)** 산출. 등급(주의)은 유지되어 FE 서사 골격과 호환.
- 회복 전 원점은 47(위험) — 첫 정리 조치(싸이월드)로 위험→주의 경계를 넘는 회복 아크가 데모에서 자연 발생.
- 정리 완주 시 상한: 유출 3건 해결(+24) + 휴면 정리(+13) + 비밀번호 교체(+8) + 세션 정리(+3) ≈ 98 → 목표 85 초과 달성 가능(FE `targetScore=85`와 호환).

## 5. 계약·경계

- `computePrivacyScore(signals, coverage)` 시그니처 유지(T1.2 고정). `ScoreSignals`는 기존 5필드 보존 + 티어 분해 필드 추가(상위호환).
- coverage 인자는 점수에 미반영(정직 표기 전용) — 시그니처 유지 목적으로만 수취.
- 비밀번호 노출 판정: `Breach.exposedFields`에 문자열 "비밀번호" 포함 여부(한글 필드값이 dummy-data 정본 계약).
- 미확인(`lastUsedAt=null`) 계정: 엔진 무감점 + coverage 하락. `/api/accounts` 표시 로직(3650일 폴백)과의 정합은 T3.4에서 검토.
